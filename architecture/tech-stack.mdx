HNU-TimeLetter is built on Next.js 16 with the App Router, TypeScript 5 in strict mode, and Tailwind CSS v4 for styling. Animation is handled by Framer Motion v12 for opening sequences, transition overlays, and card effects, while Lenis provides buttery smooth scroll on the desktop unlocked state. Global UI state lives in Zustand v5, and component primitives come from Radix UI surfaced through the Shadcn/UI pattern. Turbopack is the default bundler during development. In production the app reads CMS data from Feishu Bitable, stores binary assets in Aliyun OSS, and deploys to a self-hosted server (Vercel static export is also supported).

## External services

<CardGroup cols={2}>
  <Card title="Feishu Bitable" icon="table">
    Single source of truth for all CMS data: story records, creation board ideas, location coordinates, and contributor credits. Accessed via the Feishu Open API during sync runs.
  </Card>
  <Card title="Aliyun OSS" icon="cloud">
    Object storage for avatar images and full-size story illustrations. Attachments are downloaded from Feishu, re-uploaded to OSS, and the resulting URLs are written back to Feishu fields and persisted in the local JSON outputs.
  </Card>
  <Card title="Vercel / Self-hosted" icon="server">
    Primary deployment target is a self-hosted single-instance Node.js server so that `node-schedule` and the persistent sync job store work without a serverless timeout. Vercel static export is also supported for read-only deployments.
  </Card>
  <Card title="node-schedule v2.1.1" icon="clock">
    In-process cron scheduler for automatic periodic syncs. Defined in `src/lib/admin/scheduler.ts`. Only appropriate for the single-instance server deployment model.
  </Card>
</CardGroup>

## Build scripts

| Script | Command | Description |
|---|---|---|
| `dev` | `next dev` | Start development server with Turbopack |
| `build` | `next build` | Production build (no sync) |

<Note>
The repository root also contains `agents/`, `.opencode/agents/`, and `.trae/skills/` directories. These serve the development workflow and are **not** part of the `src/` runtime boundary. They are excluded from TypeScript compilation via `tsconfig.json`.
</Note>

## Engineering constraints

### TypeScript compile scope

`tsconfig.json` explicitly excludes the `agents/` directory. This prevents AI collaboration assets from entering the application compilation graph and interfering with `npm run build`. Any TypeScript inside `agents/` is treated as out-of-band tooling, not application code.

### Collaboration assets boundary

| Directory | Purpose | In build? |
|---|---|---|
| `src/` | Application source | Yes |
| `agents/` | AI agent definitions | No |
| `.opencode/agents/` | OpenCode skill assets | No |
| `.trae/skills/` | Trae skill assets | No |

### Local tooling

`opencode.json` in the repository root registers a project-scoped Playwright MCP server for local development assistance. It does not participate in the production build or runtime.

### Build failure safety

When running `sync-data-and-publish` jobs, the publish phase is a separate step from the sync phase. A failed `next build` must not affect the currently running production version. The recommended flow is: sync → write JSON → validate → build → health check → restart. See [Data sync system](/architecture/data-sync) for the full publish model.
